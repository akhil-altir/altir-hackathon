What we are building

A command center, not just a registration form.

The product has two equal responsibilities: make the event easy to run and make it feel alive. Participants get a simple team workspace. Admins get control. Judges get structured scoring. The office gets a TV-ready live view.

Event flow

The participant journey.

Before 1:00 PM

Form a team

A participant selects a team name, themselves, and one available partner. Once submitted, both people disappear from the available employee dropdown.

Until lock

Edit or disband

Before lock time, teams can be corrected. After 1:00 PM, only an admin can split, merge, replace, or override team membership.

Before key reveal

Submit an idea

Teams pick from the idea bank or submit a custom idea. They can change it later, but the first idea submission is required before their API key can be revealed.

2:30 PM

Unlock the API key

Each complete team receives one preloaded API key. The key stays hidden until event start and is visible only inside the team workspace.

5:00 - 5:30 PM

Submit final assets

Teams add their GitHub repo, demo video, presentation, stack tags, and final submission status. Early final submission can earn bonus event points.

5:30 - 6:30 PM

Present and score

Judges log in individually, score teams by criterion, and admins can enter or override scores with audit notes when needed.

Scoring system

Two score layers: momentum during the event, quality at the end.

Event points keep the room energized. Judge scores decide most of the outcome. The final ranking uses a configurable weighted blend, defaulting to 60% judge score and 40% event points, with admins able to tune event points between 35% and 40%.

Role map

Every role gets a clear surface.

Participant

Logs in, forms a team, submits ideas, sees guidelines, views public leaderboards, and accesses the team API key when eligible.

Team

Shared workspace for two members, including idea, key status, repo, demo, presentation, stack tags, event points, and judge summary.

Admin

Manages event settings, employees, teams, API keys, scoring rules, judges, guide content, announcements, overrides, and TV mode.

Judge

Logs in individually, reviews team details, saves draft scores, and submits structured evaluations by configured criteria.

Public viewer

Sees team list, ideas, progress, event points, countdowns, and final winners. Sensitive fields never appear here.

After time is up

A clean gallery for every final submission.

Once the submission deadline passes, or when admins manually publish it, teams can browse everyone’s work from one easy page: ideas, GitHub repos, demo videos, presentation links, stack tags, and team details.

Feature inventory

The PRD as buildable modules.

Authentication and access
  • Individual login for participants, admins, and judges.
  • Two team members use separate accounts but share one workspace.
  • Role-based permissions across private, admin, judge, and public views.
  • Public pages may be open or protected later, depending on deployment choice.
Employee and team management
  • Admin upload or manual employee entry.
  • Name, email, department, role, and eligibility status.
  • Exactly two members per team.
  • Used employees disappear from team creation dropdowns.
  • Team edits allowed until 1:00 PM, then admin override only.
Ideas and API keys
  • Teams can select from an idea bank or submit a custom idea.
  • Idea title and description are required before API key access.
  • Teams can change the idea later.
  • Admins preload API keys and assign one per complete team.
  • Keys reveal only after idea submission and configured release time.
  • Keys never appear in public views or TV mode.
Submissions and handbook
  • GitHub repository link.
  • Demo video link or upload reference.
  • Presentation link or upload reference.
  • Technology stack tags with no restrictions.
  • Post-deadline gallery for browsing all final team submissions.
  • Search and filters by team, department, idea, stack, and status.
  • Event handbook, setup guides, timeline, and starter tracks.
  • Builder handbook for Claude Code, Codex, Cursor, Replit, Lovable, and agent skill prompts.
Judging and final rankings
  • Judge accounts and per-judge score submissions.
  • Configurable judge criteria and max points.
  • Draft and submitted score states.
  • Admin entry on behalf of judges.
  • Score override with audit note.
  • Normalized final ranking with configurable judge/event weights.

Decisions remaining

Open questions for implementation.

  • Should final judge scores be visible immediately or only after admins publish winners?
  • Should the idea bank be entered manually or imported from a document?
  • Should the MVP support direct file uploads, or should teams submit links only?
